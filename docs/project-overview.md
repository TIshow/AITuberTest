# プロジェクト概要

## このプロジェクトの目的

「AI時代の教育」をテーマにした講演の冒頭で、講演者とAIアバターがリアルタイムに対話する演出を実現する。
"AI時代のことはAIに話してもらう" という体験を聴衆に見せることが目的。

ローカル環境（Mac）で動作し、本番当日に安定して確実に動かせるシステムを構築する。

---

## 想定ユースケース

| フェーズ | シナリオ |
|---|---|
| 本番前 | 講演者がローカルで対話・音声・アバターの動作確認をする |
| 本番当日 | 講演冒頭1〜3分間、講演者がマイクでAIアバターに質問し、AIが音声と動作で答える |
| 本番後（任意） | Q&Aパートで聴衆からの質問をAIアバターに問いかけるデモを追加する |

### 具体的な本番フロー（MVP想定）

1. 講演者が「では、AIに聞いてみましょう」と宣言
2. 講演者がマイクでAIアバターに質問（例: 「AI時代に教育はどう変わると思う？」）
3. AIアバターが音声と口パクで回答する
4. 講演者がさらに深掘り質問 → AIが答える（数往復）
5. 講演者が対話を締めて通常プレゼンに移行

### 対話の内容について

- テーマは「AI時代の教育」に絞る
- 講演者が事前に用意した教育関連の文献・データをRAGとして参照させる
- 雑談や無関係なトピックへの脱線は設計上防ぐ

---

## 成功条件

- **機能的成功**: 講演者の質問に対してAIアバターが音声で自然に答えられる
- **内容的成功**: 回答が教育・AI関連の文脈に沿っており、聴衆に説得力がある
- **安定性**: 本番中にフリーズ・無応答・クラッシュが起きない
- **操作性**: 講演者が喋るだけで操作できる（特別な操作不要）
- **再現性**: リハーサルと同じ品質で本番も動く

---

## 本番運用上の制約

| 制約 | 内容 |
|---|---|
| ネットワーク | 会場Wi-Fiは不安定な場合あり → **完全ローカル動作が必須** |
| 操作者 | 講演者本人が操作 → 喋るだけでよい設計にする |
| 機材 | 講演者のMacのみ → 外部サーバー・クラウドAPI依存ゼロ |
| 遅延 | 質問から回答まで数秒以内が理想（聴衆が待てる限界） |
| リカバリー | 無応答時に講演者が手動で進められること |
| RAG資料 | 講演者が用意したテキスト・PDF等をローカルに配置 |

---

## 講演用システムとしての特徴

### 「対話デモとしてのAIVTuber」

このシステムは、聴衆の前でAIの能力を見せる **ライブデモ** である。
エンターテインメント目的の配信AITuberとも、汎用チャットボットとも異なる。

| 観点 | 本システム（講演用） | 一般的な雑談AITuber |
|---|---|---|
| 主目的 | 教育×AIテーマの対話デモ | リアルタイム双方向雑談 |
| 入力 | 講演者のマイク音声 | テキスト or 音声（不特定多数） |
| 知識ソース | 講演者が用意したRAG文書 | LLMの汎用知識 |
| 話題の範囲 | AI時代の教育に限定 | 制限なし |
| 外部依存 | ゼロ（完全ローカル） | クラウドAPI依存あり |
| 操作の複雑さ | 喋るだけ | 設定・管理多数 |
| 失敗時対応 | 講演者が即座にフォローできる | 継続が望ましい |

---

## 雑談AITuberとの違い（設計思想）

**雑談AITuber** は「何でも答えられる」を目指す。
**本システム** は「テーマに沿って、確実に、聴衆の前で対話できる」を目指す。

この違いは技術選定全体に影響する：

- LLMはローカルで動作させ、外部API不要にする（Ollama等）
- RAGで話題を教育・AIテーマに絞り、回答品質を安定させる
- 音声入力はMacのマイク → ローカルSTT（Whisper等）で処理
- UIは最小化し、「マイクで話しかけるだけ」を実現する
